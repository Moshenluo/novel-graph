# 产品思路：从“一键转换”到“分步创作”

## 1. 为什么不做直接转换

小说转剧本不是简单格式转换。直接从小说正文跳到 YAML 会带来三个问题：

- 用户看不到系统如何理解章节、人物和场景，结果显得像黑箱。
- 章节识别错误时，用户只能在最终 YAML 中倒查问题。
- 剧本初稿需要创作判断，产品应该给作者可检查的中间态，而不是强行给出“完成品”。

因此产品从“一键转化器”调整为“分步创作工作台”。

## 2. 核心产品路径

当前路径分为四步：

1. 文本导入：用户通过文档导入或复制粘贴输入小说。
2. 章节确认：系统展示识别到的章节，用户先确认输入结构。
3. 剧本草案：系统展示即将写入 YAML 的场景摘要、动作节拍、对白数量和人物候选。
4. YAML 导出：用户生成、复制、下载结构化剧本 YAML。

这条路径让用户从“原文”逐步过渡到“结构化剧本”，每一步都有明确的检查点。

## 3. 产品价值

对作者：

- 快速得到可编辑初稿。
- 能在生成前看到章节与草案，减少盲目感。
- 人物共现图谱可辅助检查角色候选是否遗漏。

对评审：

- 演示路径清晰，能完整展示输入、理解、草案、导出。
- 能看到 YAML Schema 和产品工作流设计，而不只是页面按钮。
- 本地运行、不调用外部接口，复现门槛低。

## 4. 信息架构

左侧固定为流程与状态：

- 四步创作流程。
- 当前章节数量、人物候选、草案场景、YAML 状态。
- 本地处理说明。

右侧为当前步骤的主工作区：

- 文本导入页。
- 章节确认页。
- 剧本草案页。
- YAML 导出页。

## 5. 人物共现图谱定位

人物共现图谱不是“人物关系识别”，也不输出确定关系。它只展示：

- 人名候选。
- 同一章节中共同出现的线索。

它的作用是辅助作者检查角色候选和章节分布，避免改编时漏掉重要角色。

## 6. 下一步产品方向

- 章节确认页支持手动拆分/合并章节。
- 剧本草案页支持编辑场景摘要和动作节拍。
- 人名候选支持保留/忽略。
- YAML 导出页支持作品名、导出文件名和 Schema 校验。

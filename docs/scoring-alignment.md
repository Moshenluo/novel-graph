# 评分细则对齐清单

本文档用于把项目交付内容和评分细则逐项对齐，避免作品只完成代码功能，却没有把评审关注点表达清楚。

## 1. 作品完整度与创新性（40%）

评审关注：

- 产品设计是否合理。
- 功能是否完整。
- 交互是否流畅。
- 是否有新颖性和创意。

当前对齐点：

- 主流程聚焦“文本导入 -> 章节确认 -> 剧本草案 -> YAML 导出”，避免黑箱式直接转换。
- 支持两种输入方式：文档导入和复制粘贴。
- 输入区默认空白，方便评审使用自有文本复现。
- 少于 3 章时阻止生成，符合题目约束。
- 输出结构遵循独立 YAML Schema 文档，方便继续编辑和版本管理。
- 人物共现线索嵌入剧本草案阶段，展示人名候选和同章出现线索，增强改编检查能力。

待加强方向：

- 增强章节确认页，支持用户手动修正章节边界。
- 增加 YAML 结构预览或字段说明，提升可读性。
- 增加人名候选的人工保留/忽略操作，增强创作控制感。

## 2. 开发过程与质量（40%）

评审关注：

- 架构是否清晰合理。
- 代码逻辑是否健壮。
- 代码规范、可读性。
- PR 数量与质量。
- commit 分布是否合理。

当前对齐点：

- 前端核心转换逻辑在浏览器本地完成，不依赖外部接口，便于复现。
- 删除外部 API 调用和密钥配置，降低安全与部署风险。
- `npm run build` 和 `npm run lint` 通过。
- Schema、PRD、设计迭代计划、辅助功能备份文档均放入 `docs/`，交付结构清晰。
- README 明确依赖、原创功能、运行方式和演示要求。

待加强方向：

- 将 `App.tsx` 中的转换逻辑拆分为 `src/lib/screenplay.ts`，UI 与业务逻辑分离。
- 增加基础单元测试，覆盖章节识别、3 章校验、YAML 字段输出。
- 按功能拆分 PR：文档导入、YAML 转换、人物共现图谱、文档补充分别提交。
- 每个 PR 补齐功能描述、实现思路和测试方式。

## 3. 演示与表达（20%）

评审关注：

- Demo 视频是否清晰完整。
- 是否完整表达作品主要功能和效果。

建议 Demo 脚本：

1. 开场说明题目：AI 小说转剧本工具，核心输出为结构化 YAML 剧本初稿。
2. 展示空白输入态：说明支持文档导入和复制粘贴。
3. 导入或粘贴 3 章以上小说文本。
4. 展示左侧章节数量和人名候选状态变化。
5. 进入剧本草案，展示场景摘要、节拍、对白数量和人物共现线索。
6. 进入 YAML 导出，展示 `project`、`characters`、`scenes`、`beats`、`dialogue`。
7. 展示复制和下载 YAML。
8. 结尾说明本地运行、不调用外部接口、README 和 Schema 文档位置。

视频注意事项：

- 必须有声音讲解。
- 必须覆盖核心模块，不要花太多时间讲辅助功能。
- README 中放置可访问的视频链接。
- 演示文本需至少 3 章，且能生成可读 YAML。

## 4. 提交与 PR 建议

建议拆分为以下 PR：

1. `feat: 支持小说文本生成剧本 YAML`
2. `feat: 支持文档导入和空白输入态`
3. `feat: 新增人物共现图谱辅助检查`
4. `docs: 新增 YAML Schema 与设计原因`
5. `docs: 新增 PRD 和评分对齐文档`
6. `style: 优化核心转换界面视觉和对齐`

每个 PR 描述包含：

- 功能描述。
- 实现思路。
- 测试方式。
- 是否涉及第三方依赖。
- 是否复用历史代码及来源说明。

## 5. 最终交付检查

- README 完整。
- Demo 视频链接可播放。
- YAML Schema 文档存在并说明设计原因。
- PRD 和设计迭代计划存在。
- `npm install && npm run build` 可复现。
- `npm run lint` 通过。
- 仓库没有 API Key、外部接口调用或无关实验入口。
- commit 时间和 PR 记录符合比赛要求。
